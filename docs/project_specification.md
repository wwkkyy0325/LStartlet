# LStartlet 项目规范

## 核心设计原则

### 1. 高内聚低耦合 (High Cohesion, Low Coupling)

**高内聚**：
- 每个模块/组件应该专注于单一职责
- 相关的功能应该紧密组织在一起
- 避免功能分散和职责混淆

**低耦合**：
- 模块之间通过清晰定义的接口进行通信
- 减少直接依赖，优先使用依赖注入
- 避免循环依赖和过度耦合

### 2. 简化实现，降低学习成本

- **避免过度工程**：不在此处进行过于高级或复杂的实现
- **约定优于配置**：提供合理的默认值，减少用户必需的配置项
- **智能推断**：框架应尽可能自动处理常见场景，减少用户干预
- **渐进式复杂性**：简单场景简单使用，复杂场景才需要深入了解

### 3. 统一且简单的入口接口

- **避免暴露多种等效的实现模式**：防止用户困惑
- **优先推荐一种最佳实践**：而不是提供多种选择让用户纠结
- **封装复杂性**：将底层复杂实现隐藏，只暴露简洁的公共API

## 装饰器优先策略

### 装饰器作为主要推荐用法

装饰器是本框架的核心交互方式，具有以下优势：

1. **声明式编程**：代码意图清晰，易于理解
2. **非侵入性**：不需要修改原有类结构
3. **组合性强**：多个装饰器可以轻松组合使用
4. **学习成本低**：语法直观，符合Python习惯

### 使用装饰器的场景

✅ **推荐使用装饰器的情况**：
- 功能增强（如日志、缓存、权限控制）
- 元数据定义（如路由、配置、事件绑定）
- 行为拦截（如异常处理、性能监控）
- 依赖注入和生命周期管理

❌ **避免使用装饰器的情况**：
- 需要复杂的继承层次结构
- 装饰器逻辑过于复杂影响可读性
- 需要运行时动态改变行为模式

### 继承和其他实现方式

**继承的使用原则**：
- 仅在确实需要扩展基类行为时使用
- 优先考虑组合而非继承
- 继承层次不应超过2-3层
- 基类应提供清晰的抽象接口

**其他实现方式**：
- 工厂模式：用于复杂对象创建
- 策略模式：用于算法替换
- 观察者模式：用于事件通知
- 这些模式应通过装饰器或简单API封装，不直接暴露给用户

## 框架架构规范

### 分层架构

```
用户代码层
    ↓ (装饰器API)
框架公共API层  
    ↓ (内部实现)
框架核心实现层
    ↓ (基础库)
第三方依赖层
```

### 模块职责划分

- **core/**: 框架核心功能，高内聚的独立模块
- **plugin/**: 插件系统，通过装饰器注册
- **decorators/**: 装饰器实现，用户主要交互点
- **di/**: 依赖注入系统，支持装饰器参数注入

## API 设计指南

### 装饰器设计原则

1. **参数简化**：
   ```python
   # 好的设计 - 参数最少化
   @route("/api/users")
   def get_users():
       pass
   
   # 避免过度配置
   @route(path="/api/users", method="GET", auth=True, cache=True, ...)
   ```

2. **智能默认值**：
   ```python
   # 自动推断HTTP方法
   @route("/api/users")  # 自动推断为 GET
   
   # 显式指定（仅在必要时）
   @route("/api/users", methods=["POST"])
   ```

3. **链式装饰器**：
   ```python
   @cache(ttl=300)
   @auth_required
   @route("/api/data")
   def get_data():
       pass
   ```

### 错误处理规范

- **装饰器内部错误**：应提供清晰的错误信息
- **用户代码错误**：保持原始错误栈，不掩盖问题
- **配置错误**：在启动时尽早发现并报告

## 用户体验优化

### 学习路径设计

1. **入门级别**：只需了解基本装饰器用法
2. **进阶级别**：了解装饰器参数和组合使用
3. **专家级别**：理解内部机制和自定义扩展

### 文档和示例

- 每个装饰器都应有清晰的使用示例
- 提供常见场景的最佳实践
- 避免展示过于复杂的内部实现

## 实现约束

### 技术约束

- **Python 版本**：保持兼容性，避免使用过于新潮的特性
- **依赖管理**：最小化第三方依赖，核心功能尽量自包含
- **性能考虑**：装饰器开销应尽可能小，避免运行时反射

### 代码质量

- **类型注解**：全面使用类型提示，提高IDE支持
- **单元测试**：每个装饰器都有对应的测试用例
- **代码覆盖率**：核心功能保持高测试覆盖率

## 扩展性考虑

虽然强调简化，但仍需考虑扩展性：

1. **插件系统**：通过装饰器注册插件
2. **钩子机制**：提供生命周期钩子供扩展
3. **配置覆盖**：允许高级用户覆盖默认行为

但这些扩展机制本身也应遵循简化原则，不应该增加初学者的学习负担。

## 总结

LStartlet 框架的核心哲学是：**让简单的事情保持简单，让复杂的事情成为可能但不强制**。

- **装饰器是首选**：90%的场景应该通过装饰器解决
- **继承是备选**：只有在装饰器无法满足需求时才考虑
- **简化是目标**：不断审视API，删除不必要的复杂性
- **智能是手段**：通过合理的默认值和自动推断减少用户配置

通过遵循这些规范，我们能够构建一个既强大又易用的中量级Python框架，真正降低用户的使用门槛和学习成本。
