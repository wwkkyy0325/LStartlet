# LStartlet 项目规范

## 核心设计原则

### 1. 高内聚低耦合 (High Cohesion, Low Coupling)

**高内聚**：
- 每个模块/组件应该专注于单一职责
- 相关的功能应该紧密组织在一起
- 避免功能分散和职责混淆

**低耦合**：
- 模块之间通过清晰定义的接口进行通信
- 减少直接依赖，优先使用依赖注入
- 避免循环依赖和过度耦合

### 2. 简化API，降低学习成本

- **仅23个核心API**：大幅减少公共API数量
- **避免过度工程**：不在此处进行过于高级或复杂的实现
- **约定优于配置**：提供合理的默认值，减少用户必需的配置项
- **智能推断**：框架应尽可能自动处理常见场景，减少用户干预
- **渐进式复杂性**：简单场景简单使用，复杂场景才需要深入了解

### 3. 全自动化，零配置

- **依赖注入自动化**：自动解析和注入依赖，无需手动注册
- **生命周期管理自动化**：自动调用生命周期方法，无需手动触发
- **事件系统自动化**：自动路由事件到订阅者，无需手动发布
- **配置管理自动化**：自动保存和加载配置，支持自动保存
- **目录管理自动化**：自动创建应用目录和子目录
- **日志配置自动化**：自动配置日志输出和文件轮转

### 4. 统一且简单的入口接口

- **避免暴露多种等效的实现模式**：防止用户困惑
- **优先推荐一种最佳实践**：而不是提供多种选择让用户纠结
- **封装复杂性**：将底层复杂实现隐藏，只暴露简洁的公共API

## 核心API设计

### 23个核心API

LStartlet 仅提供23个核心API，大幅降低学习成本和使用复杂度：

#### 依赖注入（2个）

1. **`inject`** - 依赖注入函数
   - 标记需要注入的属性或参数
   - 支持类型注解自动推断
   - 在框架启动时自动解析依赖

2. **`Service`** - 服务装饰器
   - 自动注册服务到DI容器
   - 支持单例和多实例模式
   - 支持自动启动服务

#### 事件系统（3个）

3. **`Event`** - 事件基类
   - 所有自定义事件必须继承此类
   - 支持dataclass自动生成初始化方法
   - 支持同步和异步发布

4. **`publish_event`** - 发布事件函数
   - 向事件总线发布事件
   - 支持同步和异步模式
   - 自动路由到订阅者

5. **`subscribe_event`** - 订阅事件函数
   - 为指定事件类型注册处理器
   - 支持条件过滤
   - 支持多个订阅者

#### 生命周期管理（4个）

6. **`Init`** - 初始化装饰器
   - 在组件初始化后触发
   - 支持条件执行和优先级控制
   - 用于初始化资源

7. **`Start`** - 启动装饰器
   - 在框架启动时触发
   - 支持条件执行和优先级控制
   - 用于启动服务

8. **`Stop`** - 停止装饰器
   - 在框架停止时触发
   - 支持条件执行和优先级控制
   - 用于停止服务和清理

9. **`Destroy`** - 销毁装饰器
   - 在组件销毁前触发
   - 支持条件执行和优先级控制
   - 用于释放资源

#### 应用信息（1个）

10. **`ApplicationInfo`** - 应用信息装饰器
    - 标记应用程序的元数据类
    - 自动创建应用目录
    - 框架自动收集和管理信息

#### 配置管理（3个）

11. **`Config`** - 配置装饰器
    - 定义配置类并自动处理验证
    - 自动推断验证规则
    - 支持类型转换和验证

12. **`get_config`** - 获取配置函数
    - 从配置管理器中读取配置
    - 支持点号分隔的嵌套键
    - 支持默认值

13. **`set_config`** - 设置配置函数
    - 向配置管理器中写入配置
    - 支持点号分隔的嵌套键
    - 自动验证和持久化

#### 日志系统（5个）

14. **`debug`** - 调试日志函数
    - 记录详细的调试信息
    - 同时输出到终端和文件
    - 支持彩色输出

15. **`info`** - 信息日志函数
    - 记录一般信息
    - 同时输出到终端和文件
    - 支持彩色输出

16. **`warning`** - 警告日志函数
    - 记录警告信息
    - 同时输出到终端和文件
    - 支持彩色输出

17. **`error`** - 错误日志函数
    - 记录错误信息
    - 同时输出到终端和文件
    - 支持彩色输出

18. **`critical`** - 严重错误日志函数
    - 记录严重错误
    - 同时输出到终端和文件
    - 支持彩色输出

#### 框架管理（2个）

19. **`start_framework`** - 启动框架函数
    - 一行代码启动框架
    - 自动处理所有复杂操作
    - 自动创建目录、配置日志、注册服务

20. **`stop_framework`** - 停止框架函数
    - 停止当前运行的框架实例
    - 自动清理资源
    - 自动触发生命周期方法

#### 装饰器工具（3个）

21. **`Interceptor`** - 拦截器装饰器
    - 通用函数拦截和修改
    - 支持拦截参数、结果和异常
    - 适用于输入验证、结果转换、异常处理

22. **`ValidateParams`** - 参数验证装饰器
    - 根据类型注解自动验证函数参数
    - 验证失败时抛出TypeError
    - 支持所有标准Python类型

23. **`Timing`** - 性能监控装饰器
    - 自动记录函数执行时间
    - 超过阈值时记录警告
    - 适用于性能分析和优化

## 装饰器优先策略

### 装饰器作为主要推荐用法

装饰器是本框架的核心交互方式，具有以下优势：

1. **声明式编程**：代码意图清晰，易于理解
2. **非侵入性**：不需要修改原有类结构
3. **组合性强**：多个装饰器可以轻松组合使用
4. **学习成本低**：语法直观，符合Python习惯

### 使用装饰器的场景

✅ **推荐使用装饰器的情况**：
- 功能增强（如日志、缓存、权限控制）
- 元数据定义（如路由、配置、事件绑定）
- 行为拦截（如异常处理、性能监控）
- 依赖注入和生命周期管理

❌ **避免使用装饰器的情况**：
- 需要复杂的继承层次结构
- 装饰器逻辑过于复杂影响可读性
- 需要运行时动态改变行为模式

### 继承和其他实现方式

**继承的使用原则**：
- 仅在确实需要扩展基类行为时使用
- 优先考虑组合而非继承
- 继承层次不应超过2-3层
- 基类应提供清晰的抽象接口

**其他实现方式**：
- 工厂模式：用于复杂对象创建
- 策略模式：用于算法替换
- 观察者模式：用于事件通知
- 这些模式应通过装饰器或简单API封装，不直接暴露给用户

## 框架架构规范

### 分层架构

```
用户代码层
    ↓ (装饰器API)
框架公共API层（13个核心API）
    ↓ (内部实现)
框架核心实现层
    ↓ (基础库)
第三方依赖层
```

### 模块职责划分

- **_di_decorator.py**: 依赖注入和生命周期管理
- **_event_decorator.py**: 事件系统和事件订阅
- **_application_info.py**: 应用信息管理
- **_config_manager.py**: 配置管理
- **_logging.py**: 日志系统
- **_path_manager.py**: 文件和目录管理

## 自动化设计

### 依赖注入自动化

1. **自动解析依赖**：
   - 检测 `inject()` 函数标记的属性
   - 从类型注解自动推断服务类型
   - 自动查找或创建依赖实例

2. **自动注册服务**：
   - 使用 `@Service` 装饰器自动注册
   - 支持单例和多实例模式
   - 支持自动启动服务

3. **自动注入依赖**：
   - 在框架启动时自动解析依赖链
   - 处理单例和多实例注入
   - 自动管理实例生命周期

### 生命周期管理自动化

1. **自动调用生命周期方法**：
   - `@Init` 方法在对象创建后自动调用
   - `@Start` 方法在初始化后自动调用
   - `@Stop` 方法在应用停止时自动调用
   - `@Destroy` 方法在对象销毁前自动调用

2. **自动管理生命周期顺序**：
   - 按照正确的顺序调用生命周期方法
   - 处理依赖关系
   - 确保资源正确释放

### 事件系统自动化

1. **自动路由事件**：
   - 使用 `publish_event()` 发布事件
   - 自动路由到所有订阅者
   - 支持同步和异步模式

2. **自动事件订阅**：
   - 使用 `subscribe_event()` 订阅事件
   - 支持条件过滤
   - 支持多个订阅者

3. **自动事件命名空间**：
   - 使用应用名称作为命名空间
   - 避免事件冲突
   - 支持跨应用事件

### 配置管理自动化

1. **自动验证配置**：
   - 使用 `@Config` 装饰器定义配置类
   - 自动推断验证规则
   - 自动进行类型转换和验证

2. **自动保存配置**：
   - 使用 `set_config()` 设置配置时自动保存
   - 支持点号分隔的嵌套键
   - 自动持久化到配置文件

3. **自动加载配置**：
   - 使用 `get_config()` 获取配置
   - 支持点号分隔的嵌套键
   - 支持默认值

### 日志系统自动化

1. **自动配置日志**：
   - 使用 `start_framework()` 自动配置日志系统
   - 自动创建日志文件
   - 自动配置日志格式和级别

2. **自动日志轮转**：
   - 每天午夜自动轮转
   - 自动删除旧日志（30天后）
   - 支持自定义轮转策略

3. **自动日志输出**：
   - 使用 `debug()`, `info()`, `warning()`, `error()`, `critical()` 记录日志
   - 同时输出到终端和文件
   - 终端支持彩色输出

### 目录管理自动化

1. **自动创建目录**：
   - 使用 `@ApplicationInfo` 装饰器自动创建应用目录
   - 自动创建子目录（logs、cache、data）
   - 支持自定义目录结构

2. **自动管理文件**：
   - 自动处理文件路径
   - 自动创建嵌套目录
   - 支持文件操作

## API 设计指南

### 装饰器设计原则

1. **参数简化**：
   ```python
   # 好的设计 - 参数最少化
   @Service(singleton=True, auto_start=True)
   class DatabaseService:
       pass
   
   # 避免过度配置
   @Service(singleton=True, auto_start=True, priority=1, lazy=True, ...)
   ```

2. **智能默认值**：
   ```python
   # 自动推断事件类型
   class MyEvent(Event):
       message: str
   
   publish_event(MyEvent(message="Hello"))
   
   # 显式指定（仅在必要时）
   await publish_event(MyEvent(message="Hello"), async_mode=True)
   ```

3. **链式装饰器**：
   ```python
   @Init
   @Start
   class DatabaseService:
       @Init()
       def initialize(self):
           pass
       
       @Start()
       def connect(self):
           pass
   ```

### 函数设计原则

1. **参数简化**：
   ```python
   # 好的设计 - 参数最少化
   set_config("app.name", "MyApp")
   
   # 避免过度配置
   set_config(key="app.name", value="MyApp", auto_save=True, validate=True, ...)
   ```

2. **智能默认值**：
   ```python
   # 自动推断应用名称
   get_config("app.name")
   
   # 显式指定（仅在必要时）
   get_config("app.name", default="DefaultApp")
   ```

3. **标准命名**：
   ```python
   # 标准日志API
   debug("调试信息")
   info("应用启动")
   warning("警告信息")
   error("错误信息")
   critical("严重错误")
   ```

## 错误处理规范

- **装饰器内部错误**：应提供清晰的错误信息
- **用户代码错误**：保持原始错误栈，不掩盖问题
- **配置错误**：在启动时尽早发现并报告
- **依赖错误**：自动检测并报告缺失的依赖

## 用户体验优化

### 学习路径设计

1. **入门级别**：只需了解基本装饰器用法
2. **进阶级别**：了解装饰器参数和组合使用
3. **专家级别**：理解内部机制和自定义扩展

### 文档和示例

- 每个API都应有清晰的使用示例
- 提供常见场景的最佳实践
- 避免展示过于复杂的内部实现
- 提供完整的示例应用

## 实现约束

### 技术约束

- **Python 版本**：保持兼容性，避免使用过于新潮的特性
- **依赖管理**：最小化第三方依赖，核心功能尽量自包含
- **性能考虑**：装饰器开销应尽可能小，避免运行时反射

### 代码质量

- **类型注解**：全面使用类型提示，提高IDE支持
- **单元测试**：每个API都有对应的测试用例
- **代码覆盖率**：核心功能保持高测试覆盖率

## 扩展性考虑

虽然强调简化，但仍需考虑扩展性：

1. **内部函数**：提供内部函数供高级用户使用
2. **钩子机制**：提供生命周期钩子供扩展
3. **配置覆盖**：允许高级用户覆盖默认行为

但这些扩展机制本身也应遵循简化原则，不应该增加初学者的学习负担。

## 向后兼容性

### 不提供向后兼容

- 框架尚未达到内部测试阶段
- 不提供向后兼容性
- 可以随时删除或修改API
- 用户需要跟随框架更新

### API稳定性

- 一旦框架达到稳定版本，API将保持稳定
- 重大变更会提前通知
- 提供迁移指南

## 总结

LStartlet 框架的核心哲学是：**让简单的事情保持简单，让复杂的事情成为可能但不强制**。

- **装饰器是首选**：90%的场景应该通过装饰器解决
- **自动化是目标**：尽可能自动化，减少用户手动操作
- **简化是核心**：仅提供13个核心API，大幅降低学习成本
- **智能是手段**：通过合理的默认值和自动推断减少用户配置

通过遵循这些规范，我们能够构建一个既强大又易用的轻量级Python框架，真正降低用户的使用门槛和学习成本。
